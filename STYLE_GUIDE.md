# Style Guide

This guide defines general style principles for technical documentation derived from this template.

## Core principles

- Be accurate before being elegant.
- Make reader tasks and decisions visible.
- Prefer concrete wording over abstract description.
- Use active voice where it improves clarity and responsibility.
- Use consistent terminology for the same concept.
- Avoid overclaiming; state limitations and prerequisites clearly.
- Keep sections scannable with meaningful headings and lists.
- Explain why something matters when the operational consequence is not obvious.

## Bilingual writing

When documentation is bilingual, preserve meaning, task order, terminology, warnings, and verification criteria across languages. Do not let German and English versions drift into different documentation states.

## Task-oriented writing

For procedures and tutorials, make clear:

- Prerequisites.
- Starting state.
- Steps.
- Expected result.
- Verification.
- Common failure modes, if relevant.

## Conceptual writing

For concepts, make clear:

- The problem or need.
- The model or mechanism.
- Boundaries and assumptions.
- Consequences for users, admins, developers, or maintainers.

## UI and product terminology

Use the exact UI label when referring to buttons, menus, fields, options, or settings. If the UI label is unclear, explain it once and then use consistent wording.

## Screenshots and visuals

Refer to visuals in the surrounding text. A screenshot should not carry critical meaning that the text never states.

## Review checklist

Before finalizing a section, check whether it is:

- Technically correct.
- Appropriate for the audience.
- Specific enough to act on.
- Free of unnecessary jargon.
- Consistent with terminology and links elsewhere.
